# Go 后端开发规则

## 1. 总原则

这个后端项目不是只为了“能跑”，还要满足：

1. 适合学习
2. 代码易读
3. 结构清晰
4. 方便后续接前端

所以默认规则是：

- **优先可读性**
- **优先重注释**
- **优先最小正确实现**
- **优先清晰分层**

---

## 2. 注释规则（重点）

你特别说了：

> 需要重度的注释，因为不是很懂后端，需要通过这个项目学习后端

所以这个项目必须遵守下面的注释规则。

### 2.1 必须写注释的地方

以下内容必须写注释：

1. 结构体定义
2. 每个 handler
3. 每个 service 方法
4. 每个 repository 方法
5. 每个 middleware
6. 每个配置结构
7. 每个数据库模型字段中不直观的含义
8. 每个关键业务判断
9. 每个不是一眼就懂的 SQL / 查询逻辑

### 2.2 注释要解释什么

注释不能只写“做了什么”，还要解释：

1. 为什么这样做
2. 这一层的职责是什么
3. 这个字段为什么存在
4. 这个判断在业务上的意义
5. 这个函数会被谁调用

### 2.3 注释风格要求

建议：

- 对外导出类型/函数：写完整注释
- 复杂逻辑块：写段落注释
- 简单函数：写一句职责说明

示例：

```go
// UserService 负责处理用户相关的业务逻辑。
//
// 这一层不直接处理 HTTP，也不直接返回数据库原始结果，
// 而是负责把 repository 的数据组织成更符合业务语义的结果。
type UserService struct {
    repo *repository.UserRepository
}
```

## 3. 分层规则

### 3.1 Handler 层

职责：

- 解析请求参数
- 调用 service
- 返回统一响应

禁止：

- 直接写复杂业务逻辑
- 直接写数据库操作

### 3.2 Service 层

职责：

- 处理业务逻辑
- 组织多个 repository 调用
- 处理状态变更规则

禁止：

- 直接依赖 Gin Context
- 把 HTTP 细节写进 service

### 3.3 Repository 层

职责：

- 处理数据库读写
- 封装查询

禁止：

- 写 HTTP 逻辑
- 写前端 DTO 拼装逻辑

---

## 4. Go 代码规则

1. 所有代码使用 Type-safe Go 写法
2. 不要滥用空接口
3. 不要用过度炫技的写法
4. 保持函数短小
5. 参数和返回值尽量明确
6. 错误要向上返回，不要静默吞掉

---

## 5. 错误处理规则

1. 每个错误都要处理
2. 对用户返回友好错误信息
3. 对日志记录技术细节
4. 不要把数据库错误原样暴露给前端
5. handler 中统一映射错误响应

示例：

- 参数错误 → 400
- 未登录 → 401
- 无权限 → 403
- 资源不存在 → 404
- 服务内部错误 → 500

---

## 6. 数据库规则

1. migration 管理表结构
2. 不手工在生产环境直接改表
3. 表名使用 snake_case
4. 字段命名统一清晰
5. 建立必要索引
6. 去重规则提前设计好

---

## 7. DTO / Model 规则

要区分：

1. **数据库模型**
2. **请求 DTO**
3. **响应 DTO**

不要直接把数据库模型原样返回给前端。

---

## 8. 接口规则

### 8.1 统一响应格式

所有接口统一：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 8.2 REST 风格

优先使用 REST 风格：

- `GET /questions`
- `GET /questions/:id`
- `POST /questions/:id/favorite`
- `DELETE /questions/:id/favorite`

---

## 9. 日志规则

1. 使用统一 logger
2. 记录请求入口
3. 记录关键业务失败
4. 不打印密码、token
5. 错误日志包含上下文

---

## 10. 测试规则

学习型项目第一版至少补：

1. service 层测试
2. repository 层测试
3. handler 层基础测试

优先测试：

- 登录
- 题库筛选
- 收藏状态变更
- 已学习/未掌握状态变更
- 浏览/练习记录增删

---

## 11. 命名规则

### 文件名

- `user_handler.go`
- `question_service.go`
- `question_repository.go`

### 结构体名

- `UserHandler`
- `QuestionService`
- `QuestionRepository`

### 方法名

- `GetQuestionList`
- `MarkQuestionLearned`
- `ToggleQuestionReview`

---

## 12. 不要做的事

1. 不要一开始就做微服务
2. 不要一开始就上复杂 DDD
3. 不要一开始就加消息队列
4. 不要一开始就搞非常重的权限系统
5. 不要为了“架构高级”而让学习成本暴涨

---

## 13. 当前项目的推荐后端风格

这个项目建议采用：

- 单体后端
- 清晰分层
- 重注释
- 先 REST，再考虑高级能力

也就是说：

**先把一个“容易看懂、容易维护、容易学”的 Go 后端做出来。**

---

## 14. 给 AI/自动生成代码的额外规则

如果后续你让 AI 帮你写后端代码，必须遵守：

1. 每个新增文件要写文件职责说明
2. 每个导出结构体、方法必须写注释
3. 每个关键逻辑块写解释性注释
4. 不允许省略错误处理
5. 不允许直接跳过分层把逻辑堆在 handler
6. 不允许为了省事不写 DTO
7. 不允许为了省事把数据库模型直接返回前端

---

## 15. 当前最重要的一条

这个 Go 后端项目的规则里，最重要的是：

> **代码必须写得像“给未来的自己教学”一样。**

也就是：

- 代码本身清晰
- 注释说明清楚
- 每层职责明确
- 改动容易追踪
