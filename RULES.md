# 前端面试 App 开发规则（RULES）

## 1. 核心原则

1. 当前阶段以前端交付优先
2. 类型安全优先于开发速度捷径
3. 先复用，后抽象
4. 架构保持最小可扩展
5. UI、状态、数据访问必须分层
6. V1 不为未来不确定需求过度设计

---

## 2. 技术规则

1. 所有业务代码必须使用 TypeScript
2. 非必要情况下禁止使用 `any`
3. 优先使用 Expo 生态兼容方案
4. 优先使用项目内已有脚本和工具链
5. 已确认的技术栈不要随意替换
6. 对基础 UI 能力，优先使用成熟、稳定、兼容 Expo 的 UI 框架
7. 能通过成熟框架解决的基础 UI 问题，优先不要手写重复组件

---

## 3. 架构规则

### 3.1 分层要求

项目代码应明确划分为：

- UI 展示层
- 功能域层
- 状态层
- 数据服务层
- 共享类型 / 工具 / 主题层

### 3.2 页面规则

1. 页面负责组合和交互，不直接堆积数据实现细节
2. 页面中不得定义原始 mock 数据
3. 页面中不得直接写未来 API 的细节逻辑
4. 页面过于复杂时必须拆分组件

### 3.3 服务层规则

1. 所有数据访问统一走 services 或 query hooks
2. mock 实现与页面组件隔离
3. 后续替换真实接口时，不应大规模改动页面层
4. 必要时增加适配层处理返回数据映射

---

## 4. 状态管理规则

### 4.1 本地状态

适用于：

- 输入值
- 开关状态
- 弹窗显示
- 局部交互行为

### 4.2 全局状态

Zustand 只用于：

- 登录态
- 用户信息摘要
- App 偏好设置
- 必须跨页面共享的轻量状态

### 4.3 异步数据

TanStack Query 用于：

- 题目列表
- 题目详情
- 收藏列表查询抽象
- 后续服务端数据缓存

### 4.4 禁止事项

禁止：

- 把所有数据都塞进 Zustand
- 在展示型组件中直接发请求
- 没有原因地在 store 和 query 之间重复维护同一份数据

---

## 5. 组件规则

### 5.1 通用要求

1. 组件职责必须单一明确
2. props 必须显式定义类型
3. 出现重复 UI 模式时应抽成通用组件
4. 避免巨型组件混合展示、状态、请求逻辑
5. 优先组合而不是复杂封装

### 5.2 通用组件

建议沉淀以下基础组件：

- Button
- Input
- ScreenContainer
- Card
- Tag
- ListItem
- EmptyState
- LoadingState
- ErrorState

### 5.3 业务组件

只在某个功能域复用的组件，放在对应 feature 目录附近，不要提前全局化。

---

## 6. 样式规则

1. 颜色、间距、字号、圆角、阴影统一走 theme token
2. 不允许在多个页面散落重复硬编码样式值
3. 布局必须优先适配移动端
4. 视觉层级必须清晰
5. 架构上预留深色模式支持
6. 风格保持现代、简洁、克制
7. 基础输入框、按钮、卡片、列表、弹层等优先复用 UI 框架组件

### 6.1 禁止魔法样式值

不要随意在页面中散落：

- 任意间距值
- 任意圆角值
- 任意十六进制颜色
- 无说明的字号

统一提取到主题系统。

---

## 7. 命名规则

1. 组件名使用 `PascalCase`
2. hooks 使用 `useXxx`
3. store 使用 `useXxxStore`
4. 类型名使用明确的领域语义
5. 常量命名必须有业务含义
6. 文件名应与主要导出内容或功能一致

### 避免模糊命名

不要使用：

- `data.ts`
- `utils.ts`
- `temp.ts`
- `common.ts`

推荐使用：

- `question-service.ts`
- `use-auth-store.ts`
- `question-card.tsx`

---

## 8. 类型规则

1. 公共领域类型统一放在 `types/`
2. 仅本模块使用的类型可以就近放置
3. 表单 schema 应尽量与 TS 类型保持一致
4. 可选字段必须反映真实不确定性，不是为了省事
5. 禁止依赖隐式数据结构约定

---

## 9. 表单规则

1. 非简单表单优先使用 React Hook Form
2. 校验优先使用 Zod
3. 校验逻辑集中管理
4. 表单页面必须给出明确错误提示
5. 即使是假登录，也按照真实表单结构组织代码

---

## 10. Mock 数据规则

1. mock 数据必须有明确类型
2. mock 数据结构应尽量贴近真实使用场景
3. mock 文件不承担复杂业务逻辑
4. mock service 的调用方式应尽量接近未来真实 API
5. mock 数据的 id 和结构必须稳定

---

## 11. API 预留规则

即使当前没有后端，也必须做到：

1. 服务接口定义清晰
2. 请求和响应结构可类型化
3. 异步调用边界统一
4. 必要时区分原始返回结构与 UI 消费结构
5. 页面渲染不得直接绑定未来后端的原始实现细节

---

## 12. 本地存储规则

1. 只持久化真正需要跨重启保留的数据
2. 典型可持久化内容包括：
   - 登录态
   - 收藏状态
   - 已学习状态
   - 最近练习记录
3. 存储逻辑必须集中管理
4. 不允许在零散 UI 组件中随意读写本地存储

---

## 13. 路由规则

1. 路由命名必须清晰稳定
2. 导航结构要反映真实信息架构
3. 不要引入过深、难维护的嵌套路由
4. 路由参数尽量类型化

---

## 14. 交互与体验规则

1. 每个页面都必须处理：
   - 加载态
   - 空状态
   - 错误状态
2. 主操作必须明确
3. 搜索和筛选要轻量直接
4. 微交互要短、轻、克制
5. 不要使用影响可用性的过度动画
6. 保持足够触控区域和间距

---

## 15. 代码质量规则

1. 不保留死代码
2. 不保留废弃注释代码
3. 不重复实现同类业务逻辑
4. 函数保持简洁可读
5. 避免隐藏副作用
6. 优先可读性，不追求炫技写法

### 15.1 常量规则

以下内容不得直接写魔法值：

- 存储 key
- 题目难度文案
- 页尺寸相关值
- 动画时长
- 列表限制
- 路由 key

必须提取为具名常量。

---

## 16. 错误处理规则

1. 异步逻辑必须处理失败场景
2. 面向用户的错误提示应友好、可理解
3. 底层错误细节不要直接暴露到 UI
4. 不允许静默吞错

---

## 17. 验证与测试规则

至少要验证：

1. App 可以正常启动
2. 路由跳转正常
3. mock 数据加载正常
4. 收藏流程可用
5. 已学习状态可用
6. 模拟面试流程可用

如果后续补测试：

- 优先测试工具函数、store、关键交互逻辑
- 不要优先写脆弱的纯样式快照测试

---

## 18. 提交规范

推荐 commit 前缀：

- `feat:`
- `fix:`
- `refactor:`
- `style:`
- `docs:`
- `chore:`

示例：

- `feat: 新增题库页与分类筛选`
- `fix: 修复收藏状态刷新后丢失问题`
- `refactor: 拆分题目详情页业务组件`

---

## 19. 范围控制规则

V1 阶段必须遵守：

1. 先完成可用页面
2. 先保证结构清晰
3. 不过早设计复杂后端模型
4. 不引入当前不必要的重型依赖
5. 先完成完整用户流程，再做高级功能

V1 不主动扩展以下内容：

- 聊天系统
- 社区动态
- 视频面试
- 复杂数据分析
- 管理后台

除非后续明确提出。

---

## 20. 默认决策规则

当实现方案有多个选择时，优先级如下：

1. 更简单的结构
2. 更强的类型约束
3. 更少的依赖数量
4. 更清晰的数据边界
5. 更容易替换为真实后端的方案

在引入新依赖前先判断：

- 它是否解决当前 V1 的真实问题？
- 是否兼容 Expo / RN？
- 是否符合当前项目架构？
- 现有工具是否已经足够？

如果答案不明确，则不要引入。
