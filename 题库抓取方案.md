# 题库抓取方案

## 1. 先说结论

是的，**你的题库数据大概率需要从网上抓，或者从公开可用资料做导入整理**。

但不要直接一上来大规模乱抓，建议分 3 种方案：

1. **手工整理导入**（最稳）
2. **从公开可转载来源抓取**（次优）
3. **自己构建题库后台录入**（长期最稳）

---

## 2. 推荐顺序

### 第一阶段：小规模手工整理

建议先做 100~300 道题的精品题库：

- React
- React Native
- TypeScript
- JavaScript
- 网络
- 浏览器
- 工程化

来源可以是：

- 自己整理
- 开源面试题仓库
- 公开博客中允许转载/整理的内容

优点：

- 质量高
- 可控
- 法律风险小
- 更适合第一版产品

---

## 3. 如果要抓取，必须注意

### 3.1 风险点

- robots.txt 限制
- 网站使用条款限制
- 内容版权问题
- 大规模抓取可能被封禁
- 抓下来的数据质量通常不统一

### 3.2 建议原则

1. 优先抓取 **明确公开、可引用、可整理** 的内容
2. 保留 `source_name` 和 `source_url`
3. 不要抓私有、付费、登录后内容
4. 不要高频请求，必须限速
5. 入库前必须人工审核一轮

---

## 4. 推荐数据获取方案

## 方案 A：开源仓库导入（推荐）

来源示例：

- GitHub 上公开的前端面试题仓库
- 公开 markdown / json 题库

做法：

1. 选择仓库
2. 读取 markdown / json
3. 解析题目结构
4. 分类、标签、难度标准化
5. 写入数据库

优点：

- 技术实现简单
- 结构化程度高
- 比抓网页更稳定

---

## 方案 B：网页抓取 + 清洗

适用于：

- 公开博客
- 公开知识库页面

做法：

1. 请求页面
2. 解析 HTML
3. 提取标题、正文、答案
4. 清洗样式和无关内容
5. 生成标准 Question DTO
6. 入库前去重

需要额外做：

- 限速
- 重试
- 错误日志
- 页面结构变更监控

---

## 方案 C：后台录入 + 半自动导入

长期建议：

- 做一个简单后台
- 支持手工录题
- 支持批量导入 markdown/json/csv
- 支持审核发布

这是后期最稳的方式。

---

## 5. 抓取后的标准化结构

建议所有题最终都转成统一结构：

```ts
type ImportedQuestion = {
  title: string;
  content: string;
  answer: string;
  difficulty: 'easy' | 'medium' | 'hard';
  category: string;
  tags: string[];
  sourceName?: string;
  sourceUrl?: string;
};
```

---

## 6. 去重策略

建议组合：

1. 标题去重
2. `source_url` 去重
3. `title + content hash` 去重

---

## 7. 推荐你现在怎么做

### 最适合当前项目的方案

我建议你现在不要先做复杂爬虫，而是：

1. 先确定 2~3 个公开来源
2. 先写一个 **导入脚本**
3. 先导入一批 React / RN / TS 题目
4. 后面再补抓取任务系统

也就是：

**先“导入”，后“抓取平台化”**。

---

## 8. 推荐下一步

最合理的下一步是：

1. 先确定抓取/导入来源
2. 我帮你写一份 **题库导入脚本设计**
3. 再写 **Question 数据清洗规则**

---

## 9. 结论

你题库数据基本是需要外部来源的，但建议：

- **第一版优先导入，不优先乱抓网页**
- 先小规模高质量
- 再做抓取系统

这样最稳。
